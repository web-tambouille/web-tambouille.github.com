Retour sur le WebWorkersCamp10

Catégorie : Conférence | 31 Octobre 2010 | Tags :  Javascript, Node.js, Web

Pour le premier billet de web-tambouille, rien de mieux qu’un retour de la journée WebWorkersCamp10, un Barcamp orienté Web et Javascript, organisé à La Cantine par af83. Cet évènement s’est déroulé hier, samedi 30 octobre, et nous a tenu en haleine toute la journée avec des sessions toutes plus intéressantes les unes que les autres.

Dès l’arrivée, petite session amusante de sélection d’auto-collants à se répartir entre nous : un avec notre nom et plusieurs autres avec différentes technologies. Pas de règles, on colle ce que l’on veut ! Ces auto-collants sont très variés : Firefox, Unbuntu, Cassandra, Redis ou bien encore Python. Il y en a pour tous les goûts ! Pour ma part : Scala, CouchDB, Riak, Node et Hadoop. Vraiment tout et n’importe quoi comme sélection n’est-ce pas

 

Vient le moment d’écrire au tableau les sujets dont on veut parler ou sur lesquels on souhaiterait échanger :

Et on se retrouve avec des sessions toutes plus intéressantes les unes que les autres : OAuth, Rest Mongo, RX OS, Sproutwit, Dispatch, Unhosted, MapReduce et Montgrel 2.

OAuth

Ma première session, OAuth, est un retour sur ce qu’est ce protocole d’authentification avec l’exemple de Twitter. L’idée est simple : prenons l’exemple de Facebook. Lors de notre première connexion, Facebook nous demande entre autres de saisir directement sur son site notre login/password GMail si l’on souhaite importer nos contacts GMail dans Facebook. La fonctionnalité est bonne, ce qui n’est pas le cas de l’implémentation. En effet, vous avez tout simplement fourni vos identifiants GMail à Facebook qui a donc accès à tous vos comptes Google ! Niveau trou de sécurité c’est pas mal du tout…

Alors qu’avec Twitter, une application tierce souhaitant accéder à nos données Twitter ne nous demandera jamais nos identifiants directement. En effet, l’application ne fera que demander un accès aux API Twitter. Nous sommes alors redirigés vers le site de Twitter sur une page nous demandant de nous identifier et d’autoriser ou non cette application à accéder à nos données. Certes, nous devons toujours nous authentifier mais cette fois-ci nous le faisons directement sur le site de Twitter et l’application externe ne connait absolument notre login/password. Si nous autorisons l’application tierce à accéder à nos données, Twitter renvoie un Access Token qui permettra à l’application tierce de taper sur les API Twitter. Dès lors que nous ne souhaitons refuser l’accès de nos données à cette application, il suffit de retourner sur Twitter et d’invalider l’accès (revoke access) de cette application.

Simple en pratique, les premiers retours sont que les &eacufte;changes réseaux sont plus nombreux mais surtout qu’un simple token non signé ne suffit pas pour avoir une sécurité optimale. L’idéal serait que, pour accéder aux API, ce token soit signé et qu’un transfert HTTPS se fasse. Ainsi, côté API, Twitter pourra confirmer que l’application entrante est bien celle qui possède cette clé. En effet, on s’expose toujours à un vol de clé, en l’occurrence un vol de notre Access Token. Ce sera le cas avec OAuth 2. Le mot de la fin : la sécurité, c’est surtout une histoire d’implémentation (bonne ou mauvaise) ! L’heure étant passée très vite, nous n’avons pas eu le temps de regarder de plus près OpenID, Salmon et OStatus, ce qui laisse une belle ouverture pour approfondir le sujet chez soi.

Sproutwit

Petite session de code avec la création from scratch (mais avec quelques copier-coller quand même) d’un client Web Twitter. Les technologies : framework Sproutcore, base de données SQLite le tout tournant sur du Node. Un très bonne présentation qui donne vraiment envie de regarder Sproutcore : création d’un squelette de projet en une ligne, création à la volée de méta-propriétés lors de requêtes en base, propriétés cachables avec gestion de l’éviction, composant haut niveau (liste lazy loadée selon les éléments visibles, lancement des requêtes ajax selon la scrollbar)…

Une fois le développement terminé, Sproutcore génère un fichier HTML, un JS et un CSS et toute l’application est ainsi servie en statique ! Il est possible de lazy-loader des parties du site pour ne pas avoir à charger tout d’un seul coup. A noter qu’une fois de plus Node est utilisé pour le développement, le résultat final étant une suite de fichiers statiques pouvant être déployés sur n’importe quel type de serveur. Donc toujours pas de Node en prod (même si certains évoquaient cette possibilité dans les mois à venir) ! Une session vraiment sympa.

Dispatch

Petite présentation d’une librairie en cours de développement qui permet de gérer une pile d’évènements sur un serveur NodeJS (qui pour rappel ne fonctionne qu’avec un seul thread). 2 types de piles sont implémentées : la LinearQueue et la ConcurrentQueue. Exemple avec une ConcurrentQueue :

Il est ainsi possible de contrôler les appels. Exemple : Google Translate limite les appels avec un trop grand nombre de mots à traduire ce qui peut très vite être bloquant pour une application tierce. Avec ces piles, on peut limiter les appels, les retarder ou les relancer ultérieurement. La librairie étant en développement, plusieurs idées ont surgies de toutes part comme la possibilité de faire un noeud Node spécifique qui ne contiendrait que ces piles. Ainsi, tous les autres noeuds applicatifs taperaient sur ce noeud pour récupérer les tâches à effectuer et l’instance Node de la pile n’a pas de réplication d’évènements à gérer, elle est standalone et gère elle-même ses évènements comme bon lui semble. Wait & see…

Pause pizzas / cocas  et programme de l’après-midi :

Designing REST API

Présentation au format HTML5 de Jérémi Joslin de chez Exo sur le design d’API REST avec entre autres : pourquoi faire des API, qu’est-ce qu’une ressource REST, les différents verbes HTTP, les codes d’erreurs, gestion de l’authentification, exemples d’API avec Twitter, Facebook, Google Buzz… Idéal pour découvrir ou approfondir REST.

ParisJS !!!

Il est grand temps qu’un Javascript User Group se monte sur Paris : ce sera le ParisJS ! Un google-group est en cours de création, l’objectif étant de regrouper tous les évènements Javascript à un seul endroit. Une très bonne initiative car entre les JSMeetUp, les JSCamp et les WebWorkersCamp ce n’est pas facile de s’y retrouver

20 projets en 40… en fait 20 minutes

Excellente présentation sur un format Pecha Kucha un peu adapté puisque les slides étaient d’un peu moins d’une minute (au lieu des 20 secondes idéales) :

  • validate-website, un web crawler de validation de site web
  • gource, votre repository en vidéo
  • ACK, un grep en 3 lettres
  • Dstat
  • pygments, coloration syntaxique
  • Redshift, faites du bien à vos yeux !
  • DailyStamp, tamponnez chaque jour travaillé pour obtenir un gros score
  • ElasticSearch, un search engine sur le cloud
  • ZeroMQ
  • Spore / api-description
  • Htty, une console de requêtes HTTP
  • Firesheep, no comment…
  • PNGStore, JS et CSS dans un PNG
  • PNGCrush, réduisez la taille de vos PNG
  • vapor.js, juste génial ! je fork de suite
  • TestSwarm, cloud testing multi browsers multi OS
  • Vows, CI pour Node
  • SocketIO, gestion des implémentations Web Socket pour vous (multi browsers)
  • Gollum, le wiki de github
  • Cockatoo

Pari tenu, plein de petites choses à tester.
Une dernière session idéale pour apprendre et pour rire !

Conclusion

Une journée vraiment sympa, enrichissante et très conviviale avec plein de geeks ! Et ça fait plaisir.

Un grand merci à af83 et à Ori Pekelman pour cette journée, vivement la prochaine !
