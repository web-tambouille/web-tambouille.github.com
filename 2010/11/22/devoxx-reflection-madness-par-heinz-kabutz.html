DEVOXX - Reflection Madness

Catégorie : Conférence | 22 Novembre 2010 | Tags :  Devoxx, Java

Devoxx, c’est fini. D’ailleurs, si vous voulez vivre ou revivre ce Devoxx 2010, n’oubliez pas de vous prendre un abonnement à Parley’s, le prix passe de 49€ les 6 mois à 79€ l’année. Le calcul est vite fait : on est gagnant ! Au passage, l’édition 2009 est intégralement disponible.

Et pour les sceptiques, et si mes précédents billets sur MongoDB, l’interview de Stephan Janssen et l’université Dive Into Android ne vous ont toujours pas convaincu à prendre cet abonnement, ce résumé de session pourrait bien faire pencher la balance

En effet, Heinz Kabutz, Java Champion et créateur de la newsletter Java Specialists, nous a fait une petite démonstration de réflection en Java dans cette session Reflection Madness. Peu de choses réutilisables en entreprise mais un vrai plaisir pour tout geek Java !

L’API Reflection n’est pas nouvelle puisqu’elle date de Java 1.1, celle-ci permet aux développeurs de récupérer le type à partir d’un objet, d’appeler des méthodes sur cet objet, de modifier ses attributs mais aussi de créer de nouvelles instances du type récupéré.

Les avantages : flexibilité puisque que l’on peut définir les méthodes à appeler au runtime, une certaine puissance puisque l’on peut modifier des champs privés mais aussi une solution magique pour faire toutes sortes de choses que l’on n’imagine pas pouvoir faire !

Les dangers : les outils d’analyses de code statique ne fonctionnent plus dans ce cadre, le code peut très vite devenir complexe, la compilation ne trouvera pas des erreurs pourtant évidentes (exemple d’un XML converti dynamiquement en objet par réflection), peut causer des problèmes de performances au runtime (un argument plutôt moyen car c’est de moins en moins vrai) et une mise en place pas évidente dans certains cas.

Premier exemple : que peut-on faire à partir d’une réflection sur une classe ? Et bien on peut trouver plusieurs choses :

  1. quelle est la super classe ?
  2. quelles sont les interfaces implémentées ?
  3. quels sont les champs et méthodes accessibles (direct ou des parents) ?
  4. quels sont tous les champs et méthodes définis dans la classe, incluant ce qui est privé ?
  5. quels sont les inner classes ?
  6. et quels sont les constructeurs ?

On devra ainsi passer par getField, getMethod et getConstructor et leur variante getDeclaredXXX pour accéder à chaque partie. Les attributs et méthodes privés ont besoin d’un appelle à setAccesible(true) pour fonctionner (ce qui demande un support de Security Manager).

En effet, la méthode get de la map s’appuie entre autres sur le HashCode pour vérifier que l’objet donné est bien le même que celui de la clé. OMG, ne jamais faire ça…

Autre possibilité : savoir qui nous a appelé ! Cela se fait à l’aide de l’API sun.reflect.Reflection et par la méthode getCallerClass(2). Cette méthode renvoie une Class<?> sur laquelle on peut ainsi faire un whoAmI(). L’autre possibilité est de générer une stack trace, de parser cette stack et de récupérer toute les informations. Dernier exemple de réflection pour le coup très madness : la modification de champ static final, réputé complètement non modifiable par la Spec Java ! Voici la marche à suivre sur une JVM Sun :

  1. trouver le champ par réflection,
  2. récupérer le modifiers avec Field.class.getDeclaredField(“modifiers”) puis en l’appelant sur votre champ statique par modifiersField.getInt(staticField),
  3. retirer la propiété final avec modifiersValue &= ~Modifier.FINAL et ré-injecter là avec modifiersField.setInt(staticField, modifiersValue),
  4. récupérer le FieldAccessor avec sun.reflect.ReflectionFactory et sa méthode newFieldAccessor(staticField, false),
  5. et l’utiliser pour changer la variable !

Un exemple d’utilisation donné est la gestion d’un enum dans un switch : toutes les valeurs sont bien gérées (pour le moment) et une cas par défaut lance une exception. Or, actuellement, il est impossible de rentrer dans ce cas puisque toutes nos valeurs sont correctement gérées. Il est donc possible de créer de nouvelles valeurs d’enum (pourtant static final) à des fins de tests unitaires.

Dernier cas de grosse bad practice, comme la plupart des exemples de la session d’ailleurs, utiliser la réflection pour casser le pattern singleton ! En effet, par réflection, il est possible de créer un objet sans passer par son constructeur. Mais cela passe par un bean qui étend Externalizable (et non Serializable) qui ouvre un trou de sécurité.

La conclusion de tout ça : avec cette API, nous sommes très puissants donc à utiliser avec grande sagesse car il est possible de faire de gros dégâts !!! De nombreux articles sur Java Specialists vous permettront d’aller plus loin sur le sujet.
