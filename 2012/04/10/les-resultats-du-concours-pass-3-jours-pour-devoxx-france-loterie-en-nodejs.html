Pass 3 jours pour Devoxx France : les résultats !

Catégorie : Conférence | 10Avril 2012 | Tags :  Devoxx, Node.js

A l’origine je voulais juste mettre un commentaire sur l’article du concours pour annoncer le vainqueur, mais vu la qualité des applications reçues et plus particulièrement celle du vainqueur (mais pas seulement), c’était absolument impossible de se limiter à ce petit commentaire !

Et puis après tout, c’est un concours de code : alors on va mettre les mains dedans !

Voici donc un aperçu du projet victorieux ainsi que du second : technos, modules, fonctionnement… et vous allez voir qu’avec le même serveur, on peut faire des applications diamétralement opposées ! Let’s code review !

Langage

Là je vous vois venir :

Ben Node c’est du Javascript donc les projets sont en Javascript voyons !

Oui… et non ! La preuve, le vainqueur est un projet full coffeescript ! Depuis le temps que je voulais tester le plugin Node (lancement de Node depuis l’IDE…) et l’éditeur coffeescript d’IntelliJ IDEA, en voilà une idée occasion qu’elle est bonne
Et puisqu’on est dans les plugins, n’oubliez pas d’installer aussi le plugin markdown, très bien fait et bien mieux que l’éditeur de texte standard.

Le moins que l’on puisse dire, c’est que le code est très lisible même si on ne maitrise pas coffeescript. La lecture se fait de manière fluide, cela ressemble d’ailleurs beaucoup à du Scala, c’est peut-être pour cela que j’aime beaucoup cette syntaxe. A noter que c’est le seul projet à être parti sur coffeescript, donc bravo !

Modules / Frameworks

Côté modules, on est sur du robuste et reconnu dans le monde Node avec :

  • express pour le framework web
  • socket.io pour la partie temps réel
  • underscore pour la touche fonctionnelle
  • winston pour la gestion des logs
  • pkginfo pour la lecture des infos du package.json
  • vows pour les tests

Et là aussi, le fichier package.json est parfaitement rempli (issue, versions…) faisant qu’un simple npm install suffit pour tout installer. Et comme, vous pouvez le constater, le code est testé avec Vows. Ce n’était bien sûr pas demandé mais j’avoue avoir un faible pour le code avec tests unitaires. Encore une fois du bon boulot !

Côté client, on trouve du standard et solide avec :

  • jquery (quelle surprise !)
  • moment pour la gestion des dates
  • underscore aussi côté client
  • bootstrap, le fameux composant de Twitter
  • et… CSS3 pour les roulettes ! avec des animations, des transitions, du rotate et cie, j’adore !

Je retiendrai aussi, pour le deuxième du concours, l’utilisation de Backbone.js qui m’a enfin permis de m’y mettre sérieusement : le code était une nouvelle fois très lisible et bien découpé, j’ai toutefois trouvé que le tout était assez verbeux. Je ne pense pas que ce soit le projet qui soit en cause car la plupart des exemples que j’ai ensuite parcourus ont montré le même type de code. Ce n’est pas forcément une critique, et je ne souhaite d’ailleurs pas comparer telles ou telles technos, c’est juste que pour une si petite application j’ai trouvé qu’il fallait beaucoup de lignes de code Backbone. Autre particularité : la vue est faite avec jade ! Le fichier tient en 43 lignes et encore une fois c’est hyper lisible,  vraiment un très bon choix.

Donc un grand bravo à Mathieu pour ton application, cela m’a vraiment donné envie de me remettre à Backbone et refaire du Jade ! J’avais rapidement regardé les 2 il y plus d’un an mais sans creuser plus que ça, là je vais m’y mettre sérieusement  

Archi

Le code est découpé en plusieurs parties :

  • server.coffee avec la définition des routes/pages et des évènements socket.io
  • api.coffee avec toutes les données applicatives (utilisateurs, timer…) ainsi que toutes les méthodes utilitaires et métiers de l’application
  • common.coffee avec de la configuration générale (exit, exception…)
  • logger.coffee avec toute la configuration de winston

De même, on retrouve rapidement ces petits et cela évite d’avoir un all-one file qui mélange un peu tout partout dans le fichier.

Côté client, on est sur un seul page index.html avec une popup pour donner son pseudo, puis la popup disparait pour faire apparaître les roulettes CSS3. Là encore je vous conseille de regarder le fichier car on y trouve de nombreuses bonnes pratiques Web avec entre autres les scripts en fin de pages, l’utilisation d’html5shim pour les vieux IE tout p*%£$+€, du bootstrap, bonne indentation… bon il y a encore des choses à améliorer (encoding des caractères…) mais c’est vraiment pour chipoter.

Tous les évènements et scripts de démarrage sont spécifiés dans lottery.js, pas mal de code mais les évènements sont bien nommés donc on s’y retrouve bien.

L’application est déployée sur Heroku.

Règles

Les règles de base sont respectées ainsi que les règles étendues, amusez-vous à ouvrir plusieurs navigateurs et à jouer avec plusieurs utilisateurs : tous les noms apparaissent sur tous les écrans, dès qu’un joueur à joué une case, elle est grisée pour les autres joueurs… vraiment sympa de voir le tout tourner !

Le tout avec des vrais roulettes, le blocage des paris 3 secondes avant et après, un compteur, un pavé d’informations, un pavé de résultat… et du bonus avec les liens pour réseaux sociaux, différentes infos de projet (github…) bref une vraie petite appli bien complète !

Roulement de tambouille tambours…

C’est donc pour toutes ces qualités que le projet d’Olivier Bazoud remporte ce concours ! L’application est testable sur Heroku. 

Encore bravo à tous les participants du concours, ça a été un vrai plaisir, et certaines de mes nuits en ont pâti, de faire de la code review sur vos projets : beaucoup d’idées, beaucoup de technos différentes et au final plein de nouveaux Noders !

Au plaisir de pouvoir tous vous rencontrer à Devoxx France, et si nous n’arrivons pas à nous croiser dans les couloirs n’hésitez pas à vous rendre dans la Seine C le vendredi à 15h35 pour une session Optimiser votre site Web mobile qui a été revue et améliorée pour l’occasion !
