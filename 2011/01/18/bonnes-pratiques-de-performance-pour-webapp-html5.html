Bonnes pratiques de performance pour Webapp HTML5

Catégorie : Language | 18 janvier 2011 | Tags :  Best Practices, HTML5, Javascript

En attendant que HTML5 devienne une référence incontournable en entreprise (sa standardisation n’est malheureusement pas près d’arriver), nous pouvons toujours nous former aux Web Workers, aux Web Sockets ou bien encore au Web Storage.

Pour l’instant cela reste de l’auto formation qui consiste à intégrer les bonnes pratiques d’utilisation et à identifier les difficultés qui seront souvent rencontrées dans des projets réels à grande envergure (nombre de connexions, utilisateurs simultanés ou requêtes par seconde). Bien souvent, ces besoins de performances nous en font voir des vertes et des pas mûres ! C’est ainsi que l’on expérimente les limites d’un framework, d’une architecture ou même d’un langage.

Si vous ne connaissez pas encore HTML5Rocks, le portail de Google dédié aux développeurs HTML5, il est grand temps d’aller y faire un tour ! Les tutoriaux sont nombreux et je voulais partager avec vous celui-ci : Best Practices for a Faster Web App with HTML5.

L’article met en avant 10 astuces pour avoir de bonnes performances dans une application HTML5. Encore une fois, et même si plusieurs librairies existent et nous permettent de simuler HTML5 dans d’anciens navigateurs, il sera difficile d’appliquer toutes ces bonnes pratiques en entreprise. Ce n’est toutefois pas une raison pour nous priver de ces best practices et nous allons rapidement les parcourir.

Astuce 1 : utiliser le Web Storage plutôt que les Cookies

Et un constat : toutes vos données de cookies, pour un site donné, sont envoyées dans le header de toutes vos requêtes HTTP ! Ce qui bien sûr peut avoir un impact sur le temps de réponse de vos requêtes. Une bonne pratique qui en découle est de réduire la taille de vos Cookies.

Mais grâce à HTML5, il est possible de faire mieux et d’utiliser le Web Storage avec le sessionStorage et le localStorage. Les informations sont ainsi persistées côté client pour la durée de la session ou indéfiniment et les données ne sont pas transférées lors d’une requête HTTP. Plutôt intéressant !

Astuce 2 : utiliser les transitions CSS au lieu des animations Javascript

Dans de nombreux cas, vous pouvez vous passer des animations Javascript et ainsi utiliser les transitions CSS3. Ces effets se font par l’ajout/suppression de classes sur vos éléments HTML.

Vous gagnerez en taux de transfert car globalement le code nécessaire pour effectuer un effet est plus concis en CSS3 comparer à sa version Javascript. De plus, avec les accélérations matériels au niveau GPU, vous gagnerez en fluidité.

Astuce 3 : préférer une base de données côté client plutôt que des aller-retour côté serveur

L’idée ici n’est pas de mettre votre base de données Oracle avec toutes vos données sensibles côté client (je vous voyais venir ) mais plutôt de limiter le plus possible le nombre de requêtes HTTP surtout de type XHR. C’est d’ailleurs une optimisation de performance importante recherchée pour tout site à haut trafic. Et encore une fois, le sessionStorage et le localStorage seront vos alliés.

Astuce 4 : optimisation Javascript = énormes avantages en terme de performance

L’exemple donné ici est l’objet Array qui récupère un grand nombre de méthodes à partir de Javascript 1.6. D’ailleurs, ces méthodes devraient vous rappeler l’article précédemment paru sur Web Tambouille sur la Programmation Fonctionnelle en Javascript, sauf que cette fois-ci pas de librairies externes ! Il en ressort que ces nouvelles méthodes forEach, map, filter et consorts sont plus rapides que notre bonne vieille boucle à la main for(var i=0; i<arr.length; i++). De même, il est préférable d’utiliser le parsing JSON natif à l’aide de JSON.parse(), plus rapide que n’importe quelle librairie externe.

Astuce 5 : utiliser le cache manifest et pas seulement le offline

Il y a 2 ans, Wordpress utilisa Google Gears (offline apps) pour mettre en cache plusieurs ressources de la partie admin et ainsi gagner en rapidité. Le cache manifest va aussi nous être utile.

Exemple : nous allons pouvoir traiter différemment une donnée JSON, qui évolue dans le temps, et un HTML d’enrobage, qui lui, évolue moins. Ce dernier sera caché (avec une balise spécifique) et considéré comme un template sur lequel on bouclera avec la donnée JSON pour mettre à jour une partie du site. On obtiendra alors une grosse partie du HTML qui sera caché et une autre partie plus dynamique intimement liée à de la donnée pure qui ne le sera pas.

Astuce 6 : activer l’accélération matérielle

Ce point dépendra vraiment du navigateur… mais l’idée est encore une fois de déléguer au GPU tout ce que l’on peut pour gagner en terme de temps et d’effet visuel. La petite ligne magique mais qui toutefois ne garantit rien : .hwaccel {  -webkit-transform: translateZ(0); }.

Astuce 7 : utiliser les Web Workers pour de lourdes opérations CPU

Car en effet : les Web Workers sont rapides et surtout découplés du thread de l’UI. Vous allez donc pouvoir effectuer des traitements lourds en tâche de fond et continuer de travailler sur votre interface sans être bloqué.

Astuce 8 : utiliser les attributs et input types HTML5 dans vos formulaires

En plus des existants text, password et file, HTML5 ajoute les search, tel, url, email, datetime, date, month, week, time, datetime-local, number, range and color. L’idée ici est d’utiliser les validations natives de ces champs mais aussi de pouvoir passer par les attributs required et pattern, ce dernier permettant de spécifier sa propre expression régulière de validation. De plus, si les validations ne sont pas respectées, l’envoi du formulaire est bloqué. Là aussi, pas de gestion spécifique du onclick pour bloquer selon telles ou telles conditions.

Une nouvelle fois, plus besoin de passer par une librairie externe Javascript pour faire ceci. On obtient de plus une réduction de code concernant toutes ces validations, et boum! encore un gain en termes de transfert réseau!

Astuce 9 : utiliser les effets CSS3 plutôt que des images sprites

CSS3 apporte un grand nombre de nouveautés : les linear et radial gradients, les fameux rounded corners, les masques, les transformations/rotations et bien d’autres encore ! Et, comme le dit le tutoriel, remplacer une image sprites de 2k par une centaine de bytes CSS nous fait encore gagner en transfert réseau. Une fois encore, si le navigateur ne supporte pas ces effets, des librairies comme modernizr peuvent gérer cette compatibilité descendante avec les navigateurs ne les gérant pas.

Astuce 10 : utiliser les Web Sockets au lieu de XmlHttpRequest

Les Web Sockets ont été conçues en réponse à la popularité de Comet, et les avantages des Web Sockets sont nombreux.
Prenons quelques chiffres : selon certains rapports, les Web Sockets réduisent le transfert réseau de 35% ! Et sur de gros volumes, les performances des WS sont meilleures que celle de XHR (1 à 5 fois plus rapide).

Donc si vous le pouvez, préférez l’utilisation des Web Sockets au lieu des appels XmlHttpRequest.

Conclusion

Beaucoup de bonnes pratiques dans ce tutoriel ! Quelques outils peuvent nous aider dans cette quête de la performance, je dirai même plus qu’ils sont à installer d’urgence : Page Speed, YSlow, Speed Tracer for Chrome ou DynaTrace Ajax for IE.

D’autres articles sont à venir : nous allons mettre les mains dans les Web Sockets avec Node.JS, les Web Workers (et ce ne sera pas un calcul de nombres premiers) mais aussi les transitions CSS3.

Un beau programme non ? Donc stay tuned
