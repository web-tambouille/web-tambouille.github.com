Java + Programmation fonctionnelle + Funky = FunkyJFunctional !

Catégorie : Framework | 05 Avril 2011 | Tags :  Fonctionnelle, Guava, Java

Courant de semaine dernière, alors que je faisais un important refactoring pour améliorer la lisibilité du code, je suis tombé sur une librairie au nom assez rigolo : FunkyJFunctional. Comme je suis un Funky développeur  j’ai regardé de plus près…

Comme vous le savez peut-être déjà, j’aime beaucoup la programmation fonctionnelle et le langage Scala. Et, sur ce refactoring qui concernait à la fois du code métier mais aussi des tests (tout ça en Java), j’ai pu mettre en oeuvre et utiliser de manière poussée des librairies comme Google Guava, Fest-Assert ou bien encore Mockito et plus particulièrement son package BDDMockito (pour faire du given-when-then). Cela pourra d’ailleurs faire l’objet d’un prochain article avec un exemple de code avant-après, le résultat est assez bluffant.

Bien sûr, ce n’est pas encore du DSL comme on peut en voir en Scala mais à l’aide de quelques astuces syntaxiques, ces librairies nous offrent une API fluent qui permet réellement de lire du code comme si nous lisions une phrase. Ceci est un aspect très important surtout pour les tests qui pour moi reste la documentation principale du code d’une application.

Et donc, pour pousser encore plus loin ce concept de lisibilité, voici une petite librairie qui simplifie encore plus (oui oui) certaines API Java (Runnable, Comparator), Google Guava, Swing et même Wicket ! Alors, le Funky qui est en vous est intéressé ?

Présentation

FunkyJFunctional est une librairie Java qui va nous permettre de coder de manière fonctionnelle. Java n’est bien sûr pas un langage fonctionnelle mais ce concept peut être émulé grâce à certaines astuces du langage. Des librairies comme Google Guava utilisent le concept de classe anonyme avec la définition d’un objet directement depuis une interface (Predicate<T>, Function<F, T>…) telle que :

FunkyJFunctional s’appuie lui sur le mécanisme des classes internes locales couplées au bloc d’initialisation d’instance. Et oui tout ça  et le résultat et assez surprenant puisque notre exemple ci-dessus devient :

Cette écriture rappelle bien évidemment la hidden feature n°1 de Java à savoir le double brace initialization. Au final : plus besoin de passer par une classe anonyme qui descend une méthode apply, nous avons directement des variables prédéfinies r et t qui nous permettent d’écrire directement notre code métier souhaité. On a ainsi, à la lecture, le code métier qui saute aux yeux à savoir r = t % 2 qui rappelle bien sûr ce que l’on peut voir dans d’autres langages fonctionnels (par exemple avec Scala x => x % 2).

Il est possible de définir ces classes Funky sur des :

  • Runnable et Comparator de Java,
  • Predicate, Function, Constraint et Supplier de Google Guava, 
  • ActionListener de Swing,
  • AbstractReadOnlyModel de Wicket.

Nous allons surtout regarder du côté de Java et Google Guava.

Utilisation

Comme je l’ai dit, rien de mieux de les tests pour voir comment tout fonctionne. Donc après un petit git clone et un import maven dans mon IDE préféré, me voici en train de me balader dans le code de la librairie. Une bonne documentation Javadoc ainsi que des tests pour chaque Funky classe.

Conclusion

L’idée est intéressante. Ayant l’habitude de déclarer mes Predicates et autres Comparators en constantes, que ce soit dans le service s’ils sont très spécifiques ou bien dans l’objet métier lui-même s’ils sont génériques et/ou réutilisables, j’avoue ne pas vraiment réduire et améliorer la lisibilité de mon code au moment de la composition. En revanche, on gagne un peu en lisibilité sur la création de ces FunkyObjects. Plus de méthode apply mais directement le code à base de r et de t. Plutôt agréable à l’utilisation.

Maintenant, j’avoue qu’un petit tableau de bench, à la manière de LambdaJ, serait assez intéressant surtout quand on sait que l’astuce du double brace initialization est reconnue pour être plutôt lente. Oui je sais, je pourrais le faire…

En tout cas le débat est ouvert : il y en a qui aiment, il y en a qui n’aiment pas ! Qu’en pensez-vous ?
